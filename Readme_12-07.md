Задание 1
Преимущества использования масштабирования
1. Активный master‑сервер и пассивный репликационный slave‑сервер:

Простая архитектура. Система состоит всего из двух узлов, что упрощает проектирование и развёртывание.

Легче в управлении и поддержке. Минимальное количество компонентов снижает сложность администрирования: проще настраивать, мониторить и обновлять.

Меньше точек отказа. При корректной настройке failover (автоматического переключения на резерв) система сохраняет работоспособность при сбое master‑сервера. Пассивный slave может быстро взять на себя роль нового master.

2. Master‑сервер и несколько slave‑серверов:

Распределение нагрузки на чтение между slave‑серверами. Запросы на чтение (SELECT) можно направлять на любой из slave‑серверов, снижая нагрузку на master, который обрабатывает только операции записи (INSERT, UPDATE, DELETE).

Больше производительности. За счёт распределения операций чтения достигается значительное увеличение общей пропускной способности системы, особенно для приложений с преобладанием запросов на чтение.

Гибкость системы. Slave‑серверы можно размещать географически ближе к пользователям в разных регионах, сокращая время ответа (latency).

Резервирование. Выход из строя одного или нескольких slave‑серверов не приводит к остановке системы. Остальные реплики продолжают обслуживать запросы на чтение, а master остаётся доступным для записи.

План шардирования базы данных
Принципы построения системы
Вертикальное шардирование (по столбцам):

Цель: Разделить таблицы на более мелкие части по столбцам, чтобы часто запрашиваемые и «лёгкие» данные хранились отдельно от редко используемых и «тяжёлых» (например, полей типа TEXT, BLOB).

Преимущества: Ускорение частых запросов за счёт уменьшения размера строк и объёма данных, считываемых с диска. Снижение нагрузки на индексы.

Реализация: Для каждой исходной таблицы создаётся 2–3 новых таблицы с общим первичным ключом (user_id, book_id, store_id).

Горизонтальное шардирование (по строкам):

Цель: Распределить строки одной таблицы по разным серверам (шардам) для параллельной обработки запросов и увеличения общей производительности.

Стратегии:

Hash‑based: Равномерное распределение данных по шардам с помощью хеш‑функции от ключа шардирования. Подходит для сценариев с равномерной нагрузкой.

Directory‑based (или Range): Данные группируются по определённому критерию (например, по региону). Требует отдельной таблицы‑справочника (directory) для маппинга ключа на шард.

Разбивка таблиц
1. Пользователи (Users)

Вертикальное шардирование:

user_tab1 (частые запросы): user_id, username, email, phone, status.

user_tab2 (редкие/тяжёлые): user_id, address, bio, created, last_login_details.

Горизонтальное шардирование: Hash‑based, ключ — user_id.

Формула: shard_id = hash(user_id) % N, где N — количество шардов.

2. Книги (Books)

Вертикальное шардирование:

book_tab1 (частые запросы): book_id, title, author, price, rating.

book_tab2 (редкие/тяжёлые): book_id, description, table_of_content, reviews.

Горизонтальное шардирование: Hash‑based, ключ — book_id.

Формула: shard_id = hash(book_id) % N.

3. Магазины (Stores)

Вертикальное шардирование:

store_tab1 (частые запросы): store_id, name, city, region, working_hours.

store_tab2 (редкие/тяжёлые): store_id, full_address, manager_contact, notes.

Горизонтальное шардирование: Directory‑based (по региону).

Данные группируются по полю region.

Используется таблица‑справочник region_shard_map (region_id → shard_id).

Пример: регионы 1–10 → Shard 1 (Москва), 11–20 → Shard 2 (СПб), 21–30 → Shard 3 (Урал).

Режимы работы серверов
Read‑Write (RW, мастер): Основной сервер в каждом шарде. Принимает все операции записи (INSERT, UPDATE, DELETE) и чтения (SELECT).

Read‑Only (RO, реплики): Пассивные копии (реплики) мастер‑сервера в каждом шарде. Используются исключительно для запросов на чтение (SELECT), что разгружает мастер. Обеспечивают отказоустойчивость: при сбое мастера одна из реплик может быть повышена до мастера.

Описание блок‑схемы
Клиент отправляет запрос через API / Роутер.

API / Роутер:

Содержит логику маршрутизации запросов.

Использует данные из Redis (кэш), где хранится таблица соответствия region_id → shard_id (region_to_shard).

На основании region_id из запроса определяет, на какой шард его направить.

Шарды (Shard 1, Shard 2, Shard 3):

Каждый шард — это отдельный кластер баз данных.

Внутри каждого шарда:

Один мастер‑сервер (RW): обрабатывает записи и чтение. Хранит данные для определённого диапазона регионов (например, 1–10 для Москвы).

Несколько реплик (RO): обрабатывают только чтение. Являются копиями мастера.

Репликация: Стрелки от мастера к репликам показывают процесс репликации данных в реальном времени.
