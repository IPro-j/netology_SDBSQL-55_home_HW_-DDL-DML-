**Задание 1**

Опишите основные преимущества использования масштабирования методами:

* активный master‑сервер и пассивный репликационный slave‑сервер;
* master‑сервер и несколько slave‑серверов.

**Активный master‑сервер и пассивный репликационный slave‑сервер:**

* Простая архитектура.
* Легче в управлении и поддержке.
* Меньше точек отказа.

**Master‑сервер и несколько slave‑серверов:**

* Нагрузка на чтение распределяется между slave‑серверами.
* Больше производительности.
* Гибкость системы — slave‑серверы можно размещать ближе к пользователям в разных территориях.
* Резервирование — при падении одного/нескольких slave‑сервера остальные продолжают обслуживать запросы.

Дайте ответ в свободной форме.

Разработайте план для выполнения горизонтального и вертикального (по столбцам) шардинга базы данных. База данных состоит из трёх таблиц:

* пользователи;
* книги;
* магазины (столбцы произвольно).

Опишите принципы построения системы и их разграничение или разбивку между базами данных. Пришлите блок‑схему, где и что будет располагаться. Опишите, в каких режимах будут работать сервера.

Пусть наши таблицы имеют следующие поля:

**Пользователи:** `user_id`, `username`, `address`, `bio`, `email`, `status`, `phone`, `created`, `last_login_details`.

**Книги:** `book_id`, `title`, `author`, `description`, `table_of_content`, `reviews`, `price`, `rating`.

**Магазины:** `store_id`, `name`, `full_address`, `city`, `region`, `working_hours`, `manager_contact`, `notes`.

**Вертикальное шардирование:** группируем таблицы по столбцам исходя из анализа частоты использования и «тяжести» хранимых данных.

**Пользователи:**
* `user_tab1`: `user_id`, `username`, `email`, `phone`, `status` (частые запросы, требуется низкая задержка).
* `user_tab2`: `user_id`, `address`, `bio`, `created`, `last_login_details` (редко читаемые/тяжёлые поля).
* Связь по ключу `user_id`.

**Книги:**
* `book_tab1`: `book_id`, `title`, `author`, `price`, `rating` (частые запросы, требуется низкая задержка).
* `book_tab2`: `book_id`, `description`, `table_of_content`, `reviews` (редко читаемые/тяжёлые поля).
* Связь по ключу `book_id`.

**Магазины:**
* `store_tab1`: `store_id`, `name`, `city`, `region`, `working_hours` (частые запросы, требуется низкая задержка).
* `store_tab2`: `store_id`, `full_address`, `manager_contact`, `notes` (редко читаемые/тяжёлые поля).
* Связь по ключу `store_id`.

**Горизонтальное шардирование:** разбиваем по строкам. Данные распределяем по разным серверам по ключу шардирования.

**Пользователи:** равномерное распределение. Стратегия — Hash‑based, ключ — `user_id` (`shard_id = hash(user_id) % N`).

**Магазины:** локализация по географии (плотность магазинов по территориям). Стратегия — Directory‑based. `region_id X ? shard_id` (маппинг в отдельной таблице `region_shard_map`).

**Книги:** равномерное распределение. Стратегия — Hash‑based, ключ — `book_id` (`shard_id = hash(book_id) % N`).

**Итого:**
* Для пользователей и книг нужна равномерная нагрузка — хеш.
* Для магазинов важна география и локальные операции — диапазон по региону.

```
[ КЛИЕНТ ]
     |
     v
[ API / РОУТЕР ] <--- тут живёт REGION_TO_SHARD (из Redis)
     |
     +---> [ SHARD 1 — Москва ]
     |        |
     |        +-- (RW) stores_core      (region_id 1–10)
     |        |    +-- stores_ops
     |        |
     |        +-- (RO) Replica-1a
     |        |
     |        +-- (RO) Replica-1b
     |
     +---> [ SHARD 2 — СПб ]
     |        |
     |        +-- (RW) stores_core      (region_id 11–20)
     |        |    +-- stores_ops
     |        |
     |        +-- (RO) Replica-2a
     |
     +---> [ SHARD 3 — Урал ]
              |
              +-- (RW) stores_core      (region_id 21–30)
              |    +-- stores_ops
              |
              +-- (RO) Replica-3a
```
